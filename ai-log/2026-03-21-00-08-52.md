## User Request

@"solution-architect (agent)" without updating docs. Does it worht to move inventory pets info to dynamodb and query it using lambda functions as api to avoid load on the main api? or it is better to keep all the data only in postgres and use just one api service in this case?

## Response Summary

Analysis-only response, no files changed. The agent evaluated splitting pet data to DynamoDB with a Lambda read API vs keeping everything in PostgreSQL. Verdict: do not split. At 20 traders polling every 5 seconds, the system generates ~80 queries/sec against a 60-row dataset that fits entirely in PostgreSQL's buffer cache — a db.t3.medium handles this at single-digit utilization. The split introduces dual-write consistency problems on trade execution (ownership change spans two stores without a shared transaction), cross-store application-side joins for every view, loss of referential integrity, and doubled operational overhead. Recommended alternative if load ever becomes a concern: in-application MemoryCache with 60-second TTL on pet data, reducing pet-related DB queries from ~2,400/min to 1/min with zero infrastructure changes.
